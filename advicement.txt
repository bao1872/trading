# 阶段事件统计计算底座项目说明

## 0. 项目定位

本项目是 `features` 统计目录下的**纯计算底座**，用于把现有因子库 `factor_lib` 和事件库 `event_lib` 扩展为“阶段事件统计引擎”。

核心目标不是直接判断“这是吸筹/洗盘/震仓”，而是：

1. 用现有 OHLCV 衍生指标构造可复现因子；
2. 用因子触发事件；
3. 对事件进行连续统计、循环计数和阶段成熟度评分；
4. 输出可供后续回测脚本使用的计算结果；
5. **本层不写数据库、不负责落库、不负责交易决策。**

最终研究假设：

> 成熟低位成本区中，经历多轮吸筹-洗盘循环后，如果出现末端强洗/假破并随后修复，则未来收益/回撤分布可能优于普通急杀、普通横盘或普通突破样本。

## 1. 现有结构评估

当前归档中已有两套基础库：

```text
factor_lib/
  registry.py                 # 因子注册表；只登记元数据与计算入口
  categories/                 # 因子分类实现
    trend.py                  # DSA趋势、枢轴编码、趋势动量一致性
    position.py               # DSA位置、距VWAP、距高低点、流动性位置
    momentum.py               # BBMACD与带宽/上下轨穿越
    volume.py                 # 成交量Z-score、量比、阶段量能CV
    structure.py              # 止损簇、支撑阻力、流动性池、趋势线
    quantity_price.py         # VSA努力-结果、强弱量价得分
    rhythm.py                 # DSA阶段K线数、阶段振幅、收益
    risk.py                   # ATR%、波动率、最大回撤、beta
    coordination.py           # 价格-量能/动量-量能/低位突破协同
    raw_features.py           # 原始DSA、BBMACD、vol_zscore等列
    fundamental.py            # 财务类因子占位/扩展

event_lib/
  registry.py                 # 事件注册表；事件只依赖因子列，不重新算因子
  states.py                   # 现有简单状态判定
  detectors/
    trend_events.py           # DSA翻转、上/下穿DSA VWAP、V反
    momentum_events.py        # BBMACD上/下轨、金叉/死叉
    volume_events.py          # 放量上涨、放量下跌、缩量、量价背离
    structural_events.py      # 止损簇突破、支撑/阻力突破
    breakout_events.py        # 价值区高/低位穿越，当前实现依赖dsa_pivot_pos_01
    composite_events.py       # 趋势+量、低点+缩量、动量+量、协同突破
    fundamental_events.py     # 业绩/现金流/ROE事件
```

### 1.1 已有能力

已有底层覆盖度较高：

| 事件研究维度 | 现有支撑 | 是否够第一版 |
|---|---|---|
| 趋势转平/趋势恢复 | DSA、BBMACD、trend_align、rhythm | 基本够 |
| 价格位置 | DSA VWAP、pivot_pos、距高低点、liquidity_range_pos | 基本够 |
| 波动/风险 | ATR%、volatility、MDD、BBMACD带宽 | 基本够 |
| 成交活跃/缩量 | vol_zscore、vol_ratio、VSA ER | 基本够 |
| 结构高低点 | DSA confirmed high/low、pivot type、stop cluster | 基本够 |
| 简单事件检测 | 趋势翻转、量能、突破、低点缩量、动量加速 | 已有，但不够阶段化 |

### 1.2 当前主要缺口

当前缺的不是单个指标，而是**阶段事件工程层**：

1. 缺趋势斜率类因子，尤其是 DSA VWAP 斜率、趋势转平强度；
2. 缺边界统一抽象：不同来源的 `下沿/中枢/上沿` 尚未统一；
3. 缺“成本区形成度/成熟度”因子；
4. 缺“吸筹-洗盘循环”的事件计数器；
5. 缺“末端强洗/震仓候选”事件链；
6. 缺“震仓修复度”事件链；
7. 缺阶段推进状态机，现有 `states.py` 仍偏简单趋势/筑底/震荡分类；
8. 缺多周期融合接口，后续应支持月线定位、周线结构、日线事件、分时确认。

## 2. 设计原则

### 2.1 不使用绝对时间阈值定义阶段

不要写死：

```text
横盘 60 天 = 吸筹
洗盘 3 周 = 洗盘
跌 8% = 震仓
```

应改为：

```text
趋势类指标转平
位置类指标反复测试/收回
波动类指标压缩/扩张
结构类指标循环次数累积
成交类指标配合/背离
```

### 2.2 阶段不是硬标签，而是连续证据累积

阶段判断应输出：

```text
cost_zone_forming_score
cost_zone_maturity_score
wash_cycle_count
final_shake_score
repair_score
failure_score
```

而不是直接输出：

```text
这是吸筹
这是震仓
```

### 2.3 后一阶段必须受前一阶段约束

核心逻辑：

```text
没有成熟成本区，就没有高置信震仓；
没有多轮洗盘循环，急杀只能算普通急杀；
没有修复，震仓只能算疑似震仓中，不能算震仓末期。
```

### 2.4 只做计算，不输出数据库

本模块只返回 DataFrame 或 Series：

```python
factors_df = compute_panel_v2(df)
events_df = detect_panel(factors_df)
stage_df = compute_stage_features(events_df)
```

不做：

```text
数据库写入
文件持久化
交易信号下单
实盘推送
```

## 6. 阶段引擎输出设计

建议 `event_lib/stage_engine.py` 提供：

```python
def compute_stage_scores(events_df, factors_df=None, config=None) -> pd.DataFrame:
    ...
```

输出列：

| 字段 | 含义 |
|---|---|
| `cost_zone_forming_score` | 成本区形成度 |
| `cost_zone_maturity_score` | 成本区成熟度 |
| `wash_cycle_count` | 有效洗盘循环数 |
| `wash_cycle_quality_score` | 最近洗盘质量 |
| `final_shake_score` | 末端强洗候选强度 |
| `repair_score` | 震仓后修复度 |
| `failure_score` | 结构失败/出货风险 |
| `stage_chain_score` | 阶段链条综合分 |
| `stage_guess` | 仅供研究展示的阶段推断 |

### 6.1 推荐阶段推断逻辑

```text
if failure_score 高:
    stage_guess = "结构失败/风险"
elif repair_score 高 and final_shake_score 高 and cost_zone_maturity_score 高:
    stage_guess = "疑似震仓末期/修复确认"
elif final_shake_score 高 and cost_zone_maturity_score 高:
    stage_guess = "疑似末端强洗/震仓中"
elif wash_cycle_count 增加 and cost_zone_forming_score 高:
    stage_guess = "成本区成熟/洗盘迭代"
elif cost_zone_forming_score 高:
    stage_guess = "成本区形成"
else:
    stage_guess = "无有效阶段结构"
```

### 6.2 注意

`stage_guess` 只是研究展示字段，不应直接作为买卖信号。

## 7. 第一版最小可行任务清单

### 第一阶段：补因子

## 9. 不建议第一版做的事

第一版不要做：

1. 不要直接训练机器学习模型；
2. 不要把阶段硬标成吸筹/洗盘/震仓；
3. 不要引入 L2、tick、主力资金净流入；
4. 不要把固定时间阈值当核心定义；
5. 不要重度依赖 SMC 黑箱指标；
6. 不要把 ZigZag 未确认拐点当实时已知；
7. 不要在事件库里重新计算因子；
8. 不要在纯计算底座里写数据库输出。

---

## 10. 一句话总结

当前 `factor_lib` 和 `event_lib` 已经具备趋势、位置、动量、量能、结构、风险的基础能力；下一步不是继续堆指标，而是补一层“阶段事件统计引擎”：用 DSA/位置边界/量价/波动/结构事件，连续统计低位成本区形成、吸筹-洗盘循环、末端强洗和修复确认，最终为回测验证“震仓末期是否有统计优势”提供纯计算结果。
