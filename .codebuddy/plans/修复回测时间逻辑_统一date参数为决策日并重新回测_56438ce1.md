## 产品概述

修复回测框架中 `--date` 参数的时间逻辑错误，统一语义为"决策日"（T日），并将买入执行时间修正为 T+1 日开盘。

## 核心功能

1. **语义修正**：将 `--date` 参数从"执行日"改为"决策日"（T日）
2. **买入执行时间修正**：买入操作从 T 日推迟到 T+1 日开盘执行
3. **卖出执行时间保持**：卖出操作在 T 日开盘执行（针对 T-1 日持有的仓位）
4. **全脚本同步**：所有相关脚本保持时间逻辑一致
5. **全量重新回测**：重新运行完整回测流程
6. **结果对比保留**：保留修改前结果用于对比分析

## 时间线定义

| 时间点 | 行为 | 说明 |
| --- | --- | --- |
| T-1日收盘后 | 生成候选池（使用T-1日数据） | 不变 |
| T日开盘前 | **决策**：决定卖出/买入 | `--date` 指向这个时间点 |
| T日开盘 | 执行卖出（针对T-1日持有的仓位） | 使用T日开盘价 |
| T+1日开盘 | 执行买入 | 使用T+1日开盘价 |


## 技术栈

- Python 3.x

## 实现方法

### 核心修改策略

1. **统一语义**：所有脚本的 `--date` 参数统一为"决策日"（T日）
2. **修正执行时间**：买入操作推迟到 T+1 日执行
3. **保持兼容性**：确保修改后的逻辑与回测引擎（`dynamic_exit_backtest_v2.py`）一致

### 关键修改点

#### 1. `08_daily_inference_report.py`（核心修改）

- **第1377行**：更新 `--date` 参数帮助信息为"决策日（T日）"
- **第1419-1426行**：将 `pending_buys_start` 的执行从 `target_date` 推迟到 `target_date + 1`
- **第1241行**：更新卖出执行日说明（T日开盘）
- **第1254行**：更新买入执行日说明（T+1日开盘）

#### 2. `06_daily_inference_replay.py`（同步修改）

- 检查并修正 `decide_daily_actions` 函数的调用逻辑
- 确保 `pending_buys` 在 T+1 日执行

#### 3. `05_daily_trading_sheet.py`（同步修改）

- 检查 `--date` 参数的使用逻辑
- 确保与时间线定义一致

#### 4. `run_daily.py`（生产入口检查）

- 检查传递给下游脚本的 `--date` 参数语义
- 确保与修改后的逻辑一致

### 性能考虑

- 修改不涉及性能瓶颈，主要是时间逻辑的修正
- 重新回测时利用现有的向量化操作（Pandas）提高效率
- 使用 `tqdm` 显示进度条，方便监控

### 避免技术债务

- 遵循现有代码模式和命名规范
- 修改仅限于时间逻辑，不改动核心算法
- 保持与 `dynamic_exit_backtest_v2.py` 的正确逻辑一致

## 实现说明

### 性能

- 重新回测预计4-6小时（全量数据）
- 使用向量化操作，避免Python级循环
- 小批量验证时使用少量数据（如1个月）快速验证

### 日志

- 复用现有日志系统
- 记录关键时间点：决策日、买入执行日、卖出执行日
- 避免记录敏感信息

### 影响控制

- 保留修改前结果（`backup_before_date_fix/` 目录）
- 小批量验证通过后再全量回测
- 对比修改前后的回测结果，分析差异原因

## 架构设计

### 系统架构

当前为单脚本流水线架构，各脚本独立运行但通过 `--date` 参数和共享数据文件协同。

### 数据流

1. 训练过程（不受影响）：`01_build_dataset.py` → `02_train_gbdt_models.py` → 生成模型
2. 推理过程（需修改）：`08_daily_inference_report.py` → 生成每日决策报告
3. 回测过程（需修改）：读取决策报告 → 模拟执行 → 计算收益

### 模块关系

- 训练模块 ← **不受影响**
- 推理模块 ← **需修改时间逻辑**
- 回测模块 ← **需重新运行**
