# 回滚方案文档

## 一、双写失败处理

| 场景 | 处理 | 责任人 |
|------|------|--------|
| 新表写入失败，旧表写入成功 | 记录错误日志，继续写旧表，告警通知 | 系统 |
| 新表写入成功，旧表写入失败 | 记录错误日志，继续写新表，告警通知 | 系统 |
| 新旧表都写入失败 | 记录错误日志，跳过该批次，人工介入 | 人工 |

## 二、新表异常退回

| 场景 | 处理 | 操作命令 |
|------|------|----------|
| 新表查询性能异常 | `factor_reader.py` 切回 `use_new_table=False`，读取旧表 | 改代码参数 |
| 新表数据不一致 | 停止新表写入，排查原因，修复后重新双写验证 | 停写 + 修复 |
| 新表分区问题 | 停止新表写入，修复分区方案，重建影子表 | 停写 + DDL |

## 三、旧表停止写入时机

1. **双写验证通过**：20 股 × 20 日双写验证通过，条数/冲突/查询耗时正常
2. **全量回补完成**：新表全量回补完成，与旧表数据一致
3. **读取端切换完成**：所有下游消费方已切换到新表读取
4. **观察期通过**：切换后观察 7 天，无异常告警
5. **旧表 rename**：`factor_value` -> `_factor_value_deprecated`，`event_trigger` -> `_event_trigger_deprecated`
6. **择机 drop**：观察 30 天后，确认无回滚需求，drop 旧表
